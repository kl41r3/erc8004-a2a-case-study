认证错误（`invalid api key 2049`）与余额错误（`insufficient_balance 1008`）响应结构不同，可以区分。最终确认：
- **正确 host**: `https://api.minimaxi.com/v1`
- **正确模型名**: `MiniMax-M2.5`

### 2.2 响应结构分析

MiniMax-M2.5 是推理模型，响应中含 `<think>...</think>` 推理块：

```json
{
  "choices": [{
    "message": {
      "content": "<think>\n推理过程...\n</think>\n\n{\"stakeholder_institution\": \"Olas\", ...}"
    }
  }],
  "usage": {
    "completion_tokens": 66,
    "reasoning_tokens": 60
  }
}
```

`max_tokens=256` 不够用——推理本身消耗 ~60 tokens，复杂 prompt 下推理可达 200+ tokens，导致实际 JSON 输出被截断为空字符串，触发 `json.JSONDecodeError`。

**修复**:
1. `max_tokens` 从 256 → **1024**
2. 在解析前用正则剥离 `<think>...</think>` 块

## 三、A2A 缺页补漏

### 问题

初次采集中 `--max-time 30` 过短，实际超时页数远超预期：
- Issue comments: 实有 26 页（2,600 条），初次只跑到第 9 页（900 条），**缺 1,698 条**
- PR review comments: 实有 12 页（1,200 条），初次跑到第 10 页（1,000 条），**缺 200 条**

### 补漏脚本（`scripts/patch_a2a_missing_pages.py`）

策略:
1. 从已知缺失页开始继续抓取，`--max-time 90`
2. 按 `comment_id` 去重，合并入原文件
3. 重新生成 `CHECKSUMS.json`

**结果**:
- A2A 讨论总记录：3,161 → **4,923 条**（+55.9%）
- 不重复贡献者：530 → **596 人**

这一差距提示：原始超时损失的数据量不容忽视，对于大型仓库必须验证分页完整性。

## 六、Limitations（本报告新增）

**L1: MiniMax-M2.5 推理模型的标注成本**
M2.5 每条记录消耗约 200–400 tokens（含推理），4,789 条约合 100–200 万 tokens。比 Haiku 等轻量模型贵 5–10 倍。考虑到研究精度要求（机构归属推断需要一定推理能力），此成本可接受，但需记录。

**L2: 机构归属依赖文本推断**
GitHub/论坛用户名通常不直接标注机构（如 `spengrah` 的机构需通过其 bio 或帖子内容推断）。LLM 推断存在误差，需对样本（建议 20 条）进行人工复核，计算 inter-rater agreement。

**L3: A2A 讨论记录性质差异**
A2A 的 4,923 条讨论记录以 PR review comments 为主（技术代码审查），而 ERC-8004 的 113 条为论坛治理讨论（开放议题辩论）。两者讨论性质不同，直接数量对比需谨慎，应区分"治理讨论"与"代码审查"两个维度分别分析。
