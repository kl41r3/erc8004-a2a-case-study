# R05 | 利益相关者画像、机构富化与关系网络

**日期**: 2026-03-13
**前置**: R04（LLM 全量标注完成）
**新增脚本**: `enrich_profiles.py`, `enrich_institutions.py`, `identify_core_contributors.py`, `build_network.py`

---

## 一、本期工作概述

R04 完成后，4,814 条标注记录的机构归属完全依赖 LLM 从文本内容（用户名 + 帖子正文）推断，存在以下缺陷：

- 用户名本身不直接携带机构信息，LLM 推断噪声大
- 同一人在论坛与 GitHub 上使用不同 ID（如 `Marco-MetaMask` vs `MarcoMetaMask`），被计为两人
- 无法区分"核心贡献者"与"偶发参与者"
- 贡献者之间的互动关系（回复链、协作 PR）未被结构化捕捉

本期实现了完整的利益相关者富化流水线，分四个步骤：

1. **`enrich_profiles.py`** — 拉取 Discourse 论坛 + GitHub 公开档案
2. **`enrich_institutions.py`** — 合并档案数据，升级机构标签，生成 per-author 档案
3. **`identify_core_contributors.py`** — 遴选核心贡献者，输出角色画像与人工复核清单
4. **`build_network.py`** — 构建可交互关系网络图（vis.js，无外部依赖）

## 三、机构富化结果（enrich_institutions.py）

### 方法论

机构来源优先级：
1. **GitHub company 字段** → 确定性规则覆盖（最可信）
2. **GitHub bio 文本** → 正则匹配机构关键词
3. **Discourse bio 文本** → 同上
4. **LLM 推断**（原始标注多数投票）→ 兜底

### 边类型说明

| 边类型 | 颜色 | 线型 | 构建依据 |
|--------|------|------|---------|
| `reply` | 深灰（#666） | 实线 | 论坛 `reply_to_post_number` 字段（直接回复） |
| `co_comment` | 浅灰（#aaa） | 虚线 | 同一 PR / Issue 下均有评论（co-participation） |

ERC-8004 网络以论坛回复链为主骨架，GitHub PR 共参为补充。A2A 网络全部为共参边（无回复链结构）。

### 修复记录

A2A 的 `issue_comment` 和 `pr_review_comment` 类型记录不含 `issue_number` / `pr_number` 字段，而是通过 `issue_url` / `pr_url` 携带线程 ID（如 `…/issues/169`）。修复方式：从 URL 字符串用正则提取数字 ID，与 `pr` / `issue` 类型记录统一到同一 key 空间，共参关系才得以正确计算（修复前：0 条边；修复后：238 条边）。

## 六、Limitations（本期新增）

**L5：档案数据覆盖率低（1.4%）**
大多数以太坊社区用户不填 Discourse bio，GitHub company 字段也多为空。目前仅 9 个账号的机构标签由档案数据确定；其余 617 个仍依赖 LLM 推断。在论文 Methods 节应说明机构归属的可靠性梯度（确定 → 推断 → 未知）。

**L6：Tehsmash 实为 Cisco 员工，LLM 标注错误**
A2A 第 4 高产贡献者 `Tehsmash`（113 条记录）在 GitHub company 字段写明 "Cisco Systems"，但 LLM 标注为 Independent。此类错误说明企业员工用私人账号参与开源项目时，LLM 无法从文本内容识别机构归属。建议对 Top 15 贡献者全部做档案核查。

**L7：角色推断为启发式规则，非 NLP**
`infer_role()` 基于记录数量、来源类型（是否有 `pr_body`、`review`）和立场分布设定规则，未使用分类模型。当前结果（Proposer / Core Developer / Reviewer / Community Voice）需人工复核，不宜直接作为论文数据。

**L8：A2A 关系网络密度过高**
Top 50 作者、238 条边，视觉上较拥挤。后续可按 institution 分 subgraph，或仅保留 weight ≥ 3 的强共参边，以突出核心协作结构。
